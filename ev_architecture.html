Human Computer Interactions · Final Project

EV Range Predictor

Architecture designed according to Parnas (1972): each module hides one design decision that is difficult or likely to change. Modules communicate only through abstract interfaces — never through shared internal representations.

Based on: D.L. Parnas — "On the Criteria To Be Used in Decomposing Systems into Modules" (1972)  +  "A Technique for Software Module Specification with Examples" (1972)

Parnas argues that the wrong way to decompose a system is to follow the flowchart of execution (Input → Process → Output). The right way is to begin with a list of difficult design decisions that are likely to change, then assign each decision to exactly one module which hides it from all other modules. Every other module interacts with it only through a clean abstract interface. Applied to the EV app: decisions like "how do we store battery cell data?", "which weather API do we use?", or "how do we calculate range?" are each hidden inside their own module.

Module 4 — Physics Baseline
Energy Consumption Model
// Hidden Design Decision

The exact physics formula used. Vehicle-specific constants (drag coefficient Cd, frontal area A, rolling resistance Crr, motor efficiency curve). Whether placeholders or real research data is used. The thermal derating function for cold batteries.

// Physics Model (Hidden Implementation)
E_consumption = F_drag + F_roll + F_slope + P_HVAC

where:
F_drag = ½ρCdAv² ÷ ηmotor   (aerodynamic drag)
F_roll = Crrmg   (rolling resistance)
F_slope = mg·sin(θ)   (gravitational potential energy)
P_HVAC = k(Tcabin − Tambient)   (heating/cooling power)

Constants (Tesla Model 3 example):
Cd = 0.23, A = 2.22 m², Crr = 0.011, ηmotor = 0.92, ρ = 1.225 kg/m³

Note: These constants are hidden inside this module. Other modules never see them.

Machine Learning Model Architecture

Exact model specification, training methodology, and self-learning feedback loop
Hybrid Physics + ML Correction Architecture
Two-stage prediction: physics baseline (deterministic) → ML error correction (learned)
Stage 1: Physics Baseline Model (Deterministic)

Purpose: Provide a reliable starting estimate using known aerodynamic and thermodynamic principles.

Range_physics = E_usable ÷ E_consumption_per_km

where:
E_usable = SoC × SoH × capacity_kWh × thermal_derating(T_battery)
E_consumption_per_km = (½ρCdAv² + Crrmg + mg·sin(θ)) ÷ 1000 + P_HVAC/v

Inputs:

  • From M1 (Battery): SoC, SoH, thermal state
  • From M2 (Weather): ambient temp, wind vector
  • From M3 (Topology): gradient profile θ(x)
  • From User: cargo mass, passenger count

Constants (vehicle-specific): Cd = 0.23, A = 2.22 m², Crr = 0.011, m_vehicle = 1847 kg, capacity = 75 kWh (Tesla Model 3 Long Range)

Why physics first? It works on day 1 with zero training data. It's interpretable. It handles edge cases (extreme cold, very heavy loads) better than pure ML.

Stage 2: ML Error Correction Layer (XGBoost Regressor)

Model Type: XGBoost (Gradient Boosted Decision Trees)

Why XGBoost?

  • Handles non-linear interactions between features (e.g., cold + highway speed + aggressive driving)
  • Works well with small training datasets (important for on-device learning)
  • Fast inference (~5ms on mobile hardware)
  • Naturally produces confidence intervals via quantile regression
  • Can run entirely on-device (no server needed)

Architecture:

  • Number of trees: 50 (lightweight for mobile)
  • Max depth: 5
  • Learning rate: 0.1
  • Objective: reg:squarederror (predict residual error)
  • Model size: ~200 KB (fits in phone memory easily)

What it learns: The residual error between physics prediction and actual outcome.

ε = Range_actual − Range_physics
Range_final = Range_physics + XGBoost(features)

The XGBoost model learns patterns like: "physics underestimates by 8% when temp <0°C and driver_aggression >1.5" or "physics overestimates by 12% on urban routes with frequent stops".

Training Methodology — The Self-Learning Loop

Initial State (Day 1): Model is untrained. XGBoost predicts residual = 0, so Range_final = Range_physics.

After Each Trip:

→
2

→
3
Compute Error

→
4

Save (features, ε) to local database (SQLite on-device).
→
5
Retrain
Retrain XGBoost on all stored samples. Update model weights.
→
6
Better Next Time
Next prediction uses improved model.

Retraining Frequency: After every trip (incremental learning). XGBoost retraining on 50-100 samples takes ~200ms on a phone CPU.

Data Storage (GDPR-compliant):

  • All trip data stored locally in SQLite database on device
  • NO personally identifiable information (no GPS tracks, no timestamps with location)
  • Only anonymised aggregate statistics: distance, energy, features, error
  • User can delete all data via M5.resetProfile()
  • Optional: user can opt-in to share anonymised samples to cloud for federated learning (improves global model)

Validation: 80/20 train/test split on stored samples. Monitor test set MAE (Mean Absolute Error) to detect overfitting.

Confidence Interval Estimation

Instead of predicting just the mean error, we train 3 XGBoost models:

  • Model 1: Predicts 5th percentile residual (optimistic case)
  • Model 2: Predicts 50th percentile residual (median case)
  • Model 3: Predicts 95th percentile residual (pessimistic case)

This is called Quantile Regression. XGBoost supports this natively with objective='reg:quantileerror'.

Range_low = Range_physics + Model1(features)
Range_mean = Range_physics + Model2(features)
Range_high = Range_physics + Model3(features)

UI displays: "280 km (250 – 310 km, 90% confident)"

The width of the interval (310 − 250 = 60 km) tells the user how uncertain the prediction is. Wide interval = unfamiliar conditions (new route, extreme weather). Narrow interval = high confidence (familiar route, typical conditions).

Solving the "72km Tesla Problem" with ML

Recall the incident: Tesla showed 72 km remaining but suddenly shut down due to cell imbalance.

Detection Logic (M6.detectCellImbalanceRisk):

  • Read M1.getCellVoltageDelta() and M1.getThermalState()
  • If cell_delta > 100 mV AND battery_temp < 0°C → HIGH risk
  • If cell_delta > 150 mV → CRITICAL risk (regardless of temp)

ML Enhancement: The XGBoost model learns that when cell_delta is high, the physics model overestimates range. It automatically applies a correction factor:

if cell_delta > 100 mV:
  correction = −0.15 × Range_physics (learned coefficient)
  Range_final = Range_physics + correction

UI shows: ⚠️ CRITICAL: Cell #7 imbalance detected. Displayed range may be unreliable. Charge immediately or risk sudden shutdown.

This is an actionable warning — not just "low battery", but a specific diagnosis with a specific recommendation.

Implementation Notes

Libraries:

  • iOS: CoreML (convert XGBoost → .mlmodel), or coremltools Python package for on-device training
  • Android: TensorFlow Lite (XGBoost → TFLite), or use xgboost4j-android library directly
  • Cross-platform: React Native + ONNX Runtime Mobile (XGBoost → ONNX)

Model Export:

xgboost_model.save_model('range_corrector.json')
Then convert to platform-specific format (CoreML / TFLite / ONNX).

Retraining on-device: Use incremental learning. Don't retrain from scratch — just update existing trees with new samples. XGBoost's xgb_model parameter allows this.

Placeholder Strategy (Early Development):

  • Collect real trip data from test drives
  • Train initial model offline

  • Enable on-device retraining in production
